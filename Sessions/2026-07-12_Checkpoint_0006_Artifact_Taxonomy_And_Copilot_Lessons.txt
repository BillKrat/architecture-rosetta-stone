2. Ownership / responsibility audit (rosetta-stone-AI, read-only).
   Bill's ask: in a well-designed framework, each responsibility should
   have exactly one owner — find violations. First attempt tried to
   offload the field-extraction to XMind's in-document Copilot chat
   panel (Bill's suggestion, since it was already open) — this is where
   the accidental-edit scare happened (see next item). After recovery,
   switched to a safe method: Bill downloaded the live document
   (`rosetta-stone-AI (1).xmind`) and it was read directly as a zip/JSON
   file via Bash/python, no further browser risk. Extracted every
   Owner/Responsibilities/Regulatory Notes field across all 7 core nodes
   plus Model's 3 deployment variants (API-service/Edge/RAG), Deployment
   Context Notes, and Diagnostics/Observability. Found 5 concrete
   single-owner violations, published as a second Artifact:
     - Diagnostics/Observability has NO Owner: field at all, despite
       being named (under 3 different names) as a collaborator by Tool,
       Model (API-service), and Model (Edge).
     - Deployment-context responsibilities (API-service/Edge/RAG) are
       modeled TWICE in the tree with two different owner schemes: once
       as Model's sub-topics (each with its own owner), and again in
       the separate "Deployment Context Notes" node under a single
       "Deployment owner (infra/devops)."
     - Provenance/grounding responsibility is claimed independently by
       three different owners: Retrieval, Model (RAG), and Deployment
       Context Notes -> RAG.
     - PII/data-handling policy is split three ways (Security, Memory,
       Tool) with no stated precedence between them.
     - Takedown/DMCA handling is worded almost verbatim in both
       Retrieval and Model (RAG)'s Regulatory Notes.
   Both diagram and audit are currently only Claude Artifact URLs (not
   committed to this repo) — see "Flag for Bill" below.

3. Accidental-edit incident and full recovery (rosetta-stone-AI).
   While trying to type a data-extraction request into the in-document
   XMind Copilot chat box, a click missed the input (likely a
   layout-reflow between screenshot and click) and the typed text landed
   on the canvas instead, merging into a stray new topic under "Agent."
   Undid with Cmd+Z. A follow-up attempt to clear leftover text in the
   chat box (triple-click + Cmd+A + Delete) appeared to also hit the
   canvas — the mindmap went briefly blank — prompting 2 more undos (5
   Cmd+Z total across the incident). Recovery was verified, not assumed:
   Bill downloaded the live document afterward, and it was diffed
   programmatically (title + notes + markers + full tree structure, via
   Python, unzipping both .xmind files and comparing content.json) against
   the frozen `Artifacts/XMind/rosetta-stone-AI.xmind` already committed
   to this repo. Result: 141/141 topics identical across every sheet —
   confirmed zero data loss or corruption. This diff-against-frozen-copy
   method is now the standing verification technique (see
   Topics/xmind-copilot-collaboration.md).

METHOD DISCOVERIES
--------------------
- The "Enter submits, don't embed real newlines" rule (previously
  scoped to the in-document Copilot chat panel only) is broader than
  first recorded — it applies to at least one other XMind AI text-entry
  surface (the "Create with AI" landing page) and possibly others.
  Treat it as a platform-wide XMind AI input quirk, not a single-widget
  quirk, until proven otherwise.
- A failed/interrupted XMind AI generation doesn't fail cleanly — it can
  silently spawn multiple junk documents and burn credits with no
  visible error. After any XMind AI prompt submission, check "All Maps"
  / "Recents" for unexpected new documents before assuming a single
  clean result (or none at all).
- When an accidental edit to a live, actively-used document is
  suspected, the fastest trustworthy verification is a structural diff
  against a frozen copy already committed to the repo (unzip both
  .xmind files as zip archives, compare content.json programmatically)
  rather than relying on screenshots or the app's own UI state, which
  can look ambiguous (e.g. a scroll-position blank screen looking like
  data loss when it wasn't).
- New standing practice (this session, Bill's instruction): checkpoint
  after each diagram/map created or attempted in XMind, not just at
  session end — treats the repo as the safety net for an app Claude is
  still learning to navigate reliably.
