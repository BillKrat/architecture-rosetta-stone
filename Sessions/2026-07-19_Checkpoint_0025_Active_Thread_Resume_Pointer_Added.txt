WORKING AGREEMENT
-----------------
New, durable, cross-repo convention for the shared `lets-code` /
`update-context` skills ā€” not scoped to this repo alone:

- `INDEX.md` now carries an **Active Thread** section, separate from
  the existing "Latest checkpoint" pointer. Active Thread names the
  actual content topic to resume and is maintained by `update-context`
  independent of checkpoint recency ā€” a run of process/infra-only
  sessions leaves it untouched rather than letting it drift or get
  buried behind unrelated checkpoint narrative.
- `update-context` (Step 4) now requires an explicit decision every
  checkpoint: rewrite Active Thread if the session materially advanced
  or shifted the content thread, or state plainly that it's being
  carried forward unchanged if the session was infra-only. Silence is
  not an option either way.
- `lets-code` (Steps 1 and 4) now reads Active Thread first and reports
  it first, ahead of whatever the latest checkpoint's own subject was.

METHOD DISCOVERIES
--------------------
- A pointer that lives only in prose (a checkpoint's own "Next
  Objective" line) is exactly as durable as the discipline of whoever
  writes the next one. Promoting it to a named, structurally-required
  field turns "remembered to carry it forward" into "had to make an
  explicit choice not to."
- Asked directly, Bill preferred the smaller, already-working fix
  (per-repo Active Thread + the existing repo-picker prompt) over a
  more ambitious global cross-repo pointer once the durability tradeoff
  was made concrete (git-tracked per-repo state vs. a local-only
  convenience cache). Recording the decision, not just the fix, so the
  global-pointer idea isn't silently re-proposed later ā€” see
  `[[project_repo_separation_decision]]`-style precedent of recording
  settled architecture choices explicitly.
