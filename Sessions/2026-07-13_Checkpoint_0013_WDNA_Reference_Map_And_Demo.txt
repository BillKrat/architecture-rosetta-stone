OBJECTIVES COMPLETED
---------------------

1. rosetta-stone-WDNA-Reference built (https://app.xmind.com/7oUcRnoy).
   Central topic "WDNA (Windows DNA)"; three tier nodes (Presentation
   Services, Business Services, Data Services) each with 2-3 example
   children; a note attached to "Credential Vault Service
   (SCM_RSAI_0001)" tying the map back to this repo's real
   Architecture/ content. Built via a mix of direct node editing,
   clipboard-paste list creation, and one full-structure Copilot
   generation request — all three methods demonstrated on camera as
   requested. Not part of the `rosetta-stone-AI`/`rosetta-stone-MCP-
   Bridge` freeze (see `Topics/bduf-freeze-and-training.md`) — a
   separate, living reference document.

2. Recovered cleanly from a real incident, mid-recording. An early
   title-rename click landed on canvas instead of the rename dialog
   (the dialog appeared late, after other actions were already taken),
   creating 5 stray nodes. Rather than fight an uncertain undo state,
   abandoned that document outright and started a fresh one — a
   deliberate choice to keep the recording moving rather than risk
   further compounding. Later in the same session, a much larger
   accidental deletion happened (a stray click selected the whole
   canvas via Cmd+A instead of the AI chat input, and a follow-up
   Delete/Backspace removed every branch, leaving only the central
   topic) — Bill reframed this live as "a good opportunity to see how
   effectively [Copilot] can populate nodes," and a single, fully-
   specified Copilot prompt rebuilt the entire structure correctly on
   the first attempt.

3. Two UI reliability fixes came directly from Bill, both confirmed
   working:
   - Selecting a node and pasting a dash-bulleted list
     (`"- Item 1\n- Item 2\n- Item 3"`) creates each line as a
     correctly-split child node in one paste — replaces an unreliable
     manual Tab/Return child-creation loop that had already caused one
     text-merging error this session.
   - When typed text visibly scrambled (word order corrupted) inside
     XMind's Note editor, Bill asked whether clicking at the first
     character and pasting instead of typing would avoid it. Tested
     directly: writing the text to the clipboard via script and pasting
     (`Cmd+V`) produced correct word order on the first try. Notably,
     Claude did not have a prior confirmed fix for this exact bug
     despite Bill recalling it from an earlier, undocumented session —
     said so plainly rather than imply otherwise, per
     `Standards/source-of-truth.md`.

4. `Topics/xmind-copilot-collaboration.md` substantially updated, not
   replaced. Per Bill's explicit request ("a context file associated
   with mindmap and referenced from our session context document"),
   folded all of this session's lessons into the existing topic file
   (already referenced from `INDEX.md`) rather than creating a new,
   competing one. New content: the title-rename-dialog-lag issue;
   Escape-cancels-a-node-edit (vs. Return-commits-it); the Tab/Return
   child-creation unreliability and the paste-list replacement; chat-
   panel typing still leaking onto canvas even with ref-based clicks;
   a refinement of the 2026-07-12 "whole-map requests don't work"
   lesson (that caution was about *ambiguous edits* to an existing map,
   not fully-specified *generation* on a blank one — the latter worked
   perfectly); the typed-text word-scrambling bug and its paste-based
   fix; and where the Notes feature actually lives (node selected ->
   top-toolbar `+` menu -> `Note` — not the right-click menu, not the
   Comments panel).

METHOD DISCOVERIES
--------------------
- Simulated typing has now caused distinct failures on at least four
  separate XMind UI surfaces this session alone (title rename, node
  text, child creation, note text) — this is strong enough evidence to
  treat clipboard paste as the *default* method for any multi-character
  text entry in this app, not a fallback reached for only after typing
  visibly fails.
- The 2026-07-12 "don't ask Copilot to edit the whole map at once"
  lesson was scoped correctly at the time but easy to over-apply — the
  actual risk factor is ambiguity (Copilot having to infer which
  existing content matches a description), not request size. A large,
  fully-specified generation request is a different, safer case, and
  this session is the first real evidence for that distinction.
- When an incident happens mid-recording, the fastest recovery isn't
  always careful diagnosis — abandoning a just-created, low-value
  document and starting fresh was faster and lower-risk than untangling
  an uncertain undo state, given nothing of real value existed in it
  yet. That calculus would be different for a document with real
  content (as reflected in the existing repo-diff-verification method
  for the frozen maps).
- Bill's own troubleshooting instincts (paste-a-list, paste-instead-of-
  type) were both correct and directly reusable — worth treating his
  in-the-moment UI suggestions as testable hypotheses to verify
  immediately, not just anecdotal tips.
