OBJECTIVES COMPLETED
---------------------
1. Bill asked for clarification on agentic vs. non-agentic applications
   before writing pseudocode, framing it as: an application/tool
   instantiates an Agent and uses it; he no longer sees Application or
   Tool as owning the decompose-and-manage-loop responsibility
   themselves. Confirmed correct against the settled cards (Application:
   present-interface/manage-session/route-and-deliver; Tool: Service
   provider, explicitly doesn't decide or initiate). Also surfaced that
   "agentic" is not itself a defined term anywhere in this project
   (checked Standards/glossary.md directly) — reframed the real question
   as "which node's settled Responsibilities does a system exercise,"
   flagged as claude-reasoning framing, not an adopted definition.

2. Pushed back on two parts of Bill's original mental model: "Agent
   spawning sub-agents that observe/decide/act" isn't part of the
   settled single-Agent-node model (flagged as an unadopted candidate,
   same status as other open QA-log candidates); "a tool might do
   similar with a smaller model" contradicts Tool's settled
   Service-provider stereotype — per RDD's classify-by-Responsibility-
   not-label principle (same lesson as the Controller/MVC mix-up from
   Model's Q&A), a Tool running its own decide-loop would actually be an
   Agent by Responsibility, nested inside a Tool-shaped boundary.

3. Bill worked a concrete example (query: "What does St Augustine say
   about forgiveness after confession in regards to reparation," tool =
   DCI, per his own DCI reading already established in the Model Q&A
   log). Two wording corrections surfaced against the settled Agent
   card: "Agent finds the tool" overstates the Responsibility, which is
   *select* (from tools already available), not *discover*; "hands
   gathered information off to the Model" undersells the Responsibility,
   which is *assemble* — combine Memory + Tool/Retrieval output + system
   instructions + current input into one prompt, not a raw forward.

4. Bill raised whether the iteration loop could be parallel rather than
   his original sub-agent framing. Established this is genuinely
   undetermined by the settled card (describes the Responsibility, not
   execution mechanics) and by RDD itself (pre-LLM, no concurrency
   treatment) — any parallel-execution design is Bill's own
   implementation choice to state explicitly, not groundable in the
   pristine doc.

5. Worked the concrete decomposition Bill needed before pseudocode:
   the Augustine query bundles two distinct doctrinal concepts (A:
   forgiveness after confession, B: reparation/satisfaction), plus an
   implicit sub-task C (whether Augustine ties A and B together) only
   knowable once A and B's results return. This grounds why the loop
   must be iterative rather than a fixed upfront plan, and makes the
   parallel-execution question concrete: A and B are independent enough
   to run concurrently, C depends on both finishing — a per-sub-task
   dependency, not an all-or-nothing choice.

6. Separately (start of session): ran the fewer-permission-prompts
   skill at Bill's request, global scope. Added 6 read-only MCP tool
   patterns (Claude-in-Chrome/Claude_Browser page-read tools) to
   ~/.claude/settings.json, which did not exist before. Nearly all
   frequent Bash commands were already covered by Claude Code's
   built-in auto-allow list; curl-to-api.github.com was deliberately
   excluded despite frequency (19 calls) because a wildcard prefix
   pattern could also match an appended -X DELETE/POST on the same
   prefix, same risk category as the skill's explicit gh-api exclusion.
   This change lives outside this repo (global Claude Code settings,
   not tracked in this repo's git history) — noted here for continuity
   only.
