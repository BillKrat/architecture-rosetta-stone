WORKING AGREEMENT
-----------------
New durable pattern, captured in full at
`Topics/pseudocode-driven-comprehension.md`: for future node Q&A, Bill
will write pseudocode himself to force real understanding of a concept
in motion, Claude reviews it with full code-review rigor (real bugs
named specifically, not softened because it's a learning exercise, not
just validated), and the corrected block becomes both the comprehension
check and the durable note — no separate summary gets written
afterward. Bill's own words: "we're not cheating by having you clean it
up - it served its purpose. This will most likely be our iterative
process." Distinguished explicitly from the existing "hold edits to the
polished bullets" rule from Checkpoint 0018 — related principle
(don't let Claude's polish substitute for Bill's own struggle), applied
to a different kind of artifact (working code exercises vs. the node
Responsibility/Collaborator bullets themselves).

OBJECTIVES COMPLETED
---------------------

1. Reviewed three iterations of Bill's own pseudocode for a document-
   upload/query flow (`prepareForUpload`, `Chunk`, `dalVectorStore`,
   `sendYourQuery`), catching real, specific defects each pass rather
   than validating: a dead conditional that could never evaluate false,
   a parameter-name mismatch, a `static class` contradicting an
   interface and constructor it had been given, an undeclared
   dependency reference, and — the substantive one — a broken pairing
   between each vector and its own source chunk of text.

2. That last finding is the session's real conceptual payoff, restated
   plainly: **a vector is only useful if whatever finds it can also
   retrieve what it was a vector of.** Bill's first draft returned the
   whole original document alongside the chunk vectors, losing the
   index-for-index link between `chunked[i]` and `embeddings[i]`.
   Walked this through concretely at query time (a matched vector index
   with no way to recover its source text) rather than only describing
   it abstractly, then showed the fix: pairing `Chunk(Text, Embedding)`
   together and carrying that pairing all the way into storage.

3. Recorded this pseudocode thread in
   `Topics/rosetta-stone-ai-node-definitions.md`'s Model Q&A log as the
   implementation-detail version of an already-logged claim (Model
   never touches the uploaded document) — useful because it's where
   that abstract claim turns into a concrete reason upload-time code has
   to be structured a specific way.

4. Bill circled back to Model directly and gave an explicit status
   read: Responsibilities are adequately understood. Of the two
   Collaborators, Agent is understood well enough to move on (queued as
   its own next dedicated Q&A). Retrieval still reads as "broadly
   ambiguous" as a Model collaborator — flagged explicitly by Bill
   rather than forced to a premature resolution; expected to clarify
   once Retrieval gets its own focused Q&A pass. Task/responsibility
   ownership (deferred at the end of Checkpoint 0018) is still open —
   this session's pseudocode work clarified the Model/Retrieval boundary
   but didn't reach that specific question.
