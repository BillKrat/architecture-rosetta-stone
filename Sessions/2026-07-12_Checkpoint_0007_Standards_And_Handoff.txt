OBJECTIVES COMPLETED
---------------------

1. Standards/ folder (identifier scheme + relationship vocabulary).
   Bill seeded a `TYPE_SCOPE_SEQ` identifier format
   (e.g. `UC_RSAI_00001`, `RD_RSAI_0001`) meant to give every artifact
   across every future project a permanent, unambiguous ID, and asked
   Claude to flush out the standard rather than take the seed literally.
   Two inconsistencies in the seed were caught and resolved before
   building: digit-width mismatch (standardized on 4 digits everywhere)
   and RD's meaning (Bill's example tagged one atomic requirement
   statement, not a whole document — split into RD = Requirement
   Definition vs. BRD = Business Requirements Document, the containing
   deliverable). Built `Standards/README.md`, `identifier-scheme.md`
   (full TYPE registry — 17 document/definition codes + 7 diagram codes
   — and a SCOPE registry with `RSAI` as the first entry), and
   `relationship-vocabulary.md` (7 starter predicates — partOf,
   realizes, satisfies, collaboratesWith, dependsOn, tracesTo,
   supersedes/supersededBy — recorded as structured YAML rather than
   only prose). The relationship vocabulary was explicitly designed
   around Bill's stated long-term vision of eventually loading every
   architecture deliverable into a triple store, SPARQL-navigable "up,
   down, and sideways" — flagged by Bill as deep backlog, not current
   work, so nothing was built toward it beyond making today's recording
   convention additive-friendly for a future export step.

2. Architecture/rosetta-stone-AI-Architecture/ scaffolded, then exercised
   for real. Following on from the prior session's XMind-vs-wiki-vs-
   markdown decision (Checkpoint 0006's tail end), Bill confirmed the
   in-repo-markdown-folder approach and asked for the appropriate name
   under the established `rosetta-stone-<Concern>-Architecture`
   convention. Built a `README.md` index plus 5 category files
   (Requirements & Use Cases, Logical Architecture, Physical & Technical
   Specs, System Policies, Operational Plans), each artifact type inside
   with a first-pass "what this collects" summary tying back to this
   session's ownership-audit findings — all placeholder pending real
   content. Later in the session, Bill asked to circle back to the
   original ownership-audit work and have it become real content rather
   than sit only in Checkpoint 0006's narrative and a Claude Artifact
   URL. Added a 6th file, `risks-and-open-issues.md` (cross-cutting, not
   one of the original 5 MSDN categories — a risk can concern any of
   them), and populated it with the audit's 5 findings as `RSK_RSAI_0001`
   through `0005`, each with a `violates` relationship pointing at one of
   two new Requirement Definitions added to
   `requirements-and-use-cases.md`: `RD_RSAI_0001` ("every Responsibility
   has exactly one Owner" — Bill's own stated assumption from earlier in
   the session, formalized rather than paraphrased) and `RD_RSAI_0002`
   ("every concern is defined in exactly one place" — Claude's synthesis
   from the pattern across the 5 findings, explicitly flagged as such
   since it wasn't Bill's direct words). Also added "Use Cases" and
   "Requirement Definitions" sections to `requirements-and-use-cases.md`
   that hadn't existed before — a gap caught while doing this work: the
   Standards TYPE registry already listed `UC` and `RD` as codes mapped
   to that category, but the Architecture file itself only had sections
   for the original 3 artifact types Bill listed (BRD, FS, US). Fixed
   for consistency rather than left silently mismatched. No Use Case was
   invented to fill the section — left genuinely empty rather than
   fabricated, since a real one should come from Phase 1 training
   producing an actual scenario.

3. Standards refined from first real use. Two additions surfaced by
   actually minting IDs rather than just designing the scheme
   abstractly: added `RSK` as a new TYPE code (Risk/Known Issue,
   cross-cutting, its own file rather than forced into one of the 5
   MSDN categories) to `identifier-scheme.md`; added `violates`/
   `satisfiedBy` as a new predicate pair to `relationship-vocabulary.md`
   (the negative counterpart to `satisfies`, needed once there were real
   findings that fail to conform to a requirement); and refined the

   several small artifacts share one category file, so the convention
   was adapted to a fenced `yaml` block under each artifact's own
   heading instead, same fields, section-scoped rather than file-scoped.

METHOD DISCOVERIES
--------------------
- Standards and their first real application should happen close
  together in time, not standard-design followed by a long gap — this
  session caught two real gaps (a missing TYPE code, a missing
  relationship predicate, and a mismatch between the Standards TYPE
  registry and the Architecture file's actual sections) specifically
  *because* real IDs got minted the same session the standard was
  written, not weeks later. A standard that's never been exercised is
  unverified.
- When two parallel sessions touch the same underlying subject
  (rosetta-stone-AI) under different working agreements (frozen vs.
  living companion), the resolution isn't to pick one agreement and
  override the other — it's to make sure each session's output lands in
  the document that agreement actually governs. Investigation output
  that never gets written back into a frozen document doesn't violate
  a freeze on that document, even if the investigation itself touched
  it.
- Cross-session handoff doesn't need a new document type if an existing
  Topics/*.md file is already what the other session reads — a pointer
  section added there, plus the real content living in exactly one
  other place, does the job without creating a duplication risk.
