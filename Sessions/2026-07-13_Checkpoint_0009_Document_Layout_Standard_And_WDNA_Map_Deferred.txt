1. Made WDNA/BDUF structure durable across sessions — via INDEX.md, not
   skills. Bill asked to "update the skills and/or context file" so this
   wouldn't need reconstructing in the coding session or any future one.
   Investigated both options before choosing: `.claude/skills/` in this
   repo is local, machine-specific tooling state, excluded from git
   entirely (`.gitignore`: "Local agent/tooling state"), and `AGENTS.md`
   explicitly mandates skills stay repo-agnostic ("do not hardcode...
   one project's folder layout unless the task explicitly requires it";
   "Reuse Standard: stay reusable across ai-skills, architecture-rosetta-stone,
   BlogAI, and vs-ms-bridge"). Putting WDNA-specific content into a
   skill file would violate that existing, deliberate design and
   wouldn't even be captured by git. `INDEX.md`, by contrast, is exactly
   what the `lets-code` skill reads in full at the start of every
   session (confirmed by reading `.claude/skills/lets-code/SKILL.md`
   directly: "Step 1 — Read INDEX.md first... provide the project
   mission, topic list... Do not open any topic file yet"), is
   git-tracked, and works identically for the coding-session thread
   since it's the same repo. Added a new "Binding standards" paragraph
   directly under INDEX.md's opening mission statement — impossible to
   skip past since the file's own rule is "read this in full" — stating
   WDNA as the terminology source of truth and pointing to
   `Standards/README.md` for the full set, rather than duplicating
   Standards' content into INDEX.md itself.

2. Standards/document-layout.md — new standard, formalizing existing
   practice. Bill's ask: establish a layout/design standard for BDUF
   artifact documents themselves if one doesn't already exist in written
   form, built to hold up against future triple-store requirements
   without needing a rewrite (triple store itself stays off the
   backlog, per Checkpoint 0007). There wasn't a written one — the
   category-file template, the per-artifact-entry template (heading +
   yaml metadata block + prose), and the contributed-source-document
   template had all been used consistently since Checkpoint 0006/0007
   but never extracted into their own Standards file. Wrote
   `Standards/document-layout.md` documenting all three patterns exactly
   as already practiced (not inventing something new), explicitly
   cross-referencing rather than duplicating `identifier-scheme.md`
   (IDs), `relationship-vocabulary.md` (cross-references, triple-store
   rationale), and `source-of-truth.md` (basis) — this file governs
   *shape*, those three govern *content rules*. Noted explicitly that a
   future visual reference map (see below) is a different kind of
   artifact that should point *into* documents laid out per this
   standard, not duplicate them.

METHOD DISCOVERIES
--------------------
- When a request could be satisfied by either editing a skill or editing
  tracked project content, check whether the skill is actually
  version-controlled and repo-specific before assuming it's the right
  target — in this repo it's neither (local-only, and deliberately
  repo-agnostic by its own charter in AGENTS.md). The project's own
  `INDEX.md` + `Topics/`/`Sessions/`/`Standards/`/`Architecture/`
  structure is the actual durable, cross-machine, cross-session
  mechanism; skills are just the automation that reads it.
- A "standard for our standards" (this session's document-layout.md) is
  worth writing down explicitly once a pattern has proven itself
  consistent across a few real files, rather than leaving it as tacit
  convention only Claude remembers between sessions — otherwise it's
  exactly the kind of undocumented convention this whole project exists
  to avoid inheriting.
