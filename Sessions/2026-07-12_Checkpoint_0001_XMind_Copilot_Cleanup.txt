OBJECTIVES COMPLETED
---------------------

1. Re-established Claude-in-Chrome connectivity to XMind Cloud.
   Following the standing lesson from 2026-07-11 ("navigate to the real
   target domain first, not a throwaway test domain"), the session's
   first navigation went to `https://www.xmind.works`. That domain
   turned out to be dead (browser error page), but the domain-lockout
   bug (see Topics/claude-in-chrome-connectivity.md) still froze it as
   the session's one allowed domain, blocking `app.xmind.com` just the
   same as a bad first-navigation choice would have. Diagnosed via the
   session's local state file
   (`~/Library/Application Support/Claude/claude-code-sessions/.../local_4fd647b4-cf05-443e-a06c-c4254791150a.json`,
   cliSessionId `259a107f-b203-409e-b9f4-b240d931cb0b`), wrote a fresh
   incident doc and a session-specific `.command` fix script under
   `Artifacts/XMind/`, and refined the standing rule: the first
   navigation must go to a domain confirmed to actually load, which for
   this project means `app.xmind.com` specifically (not
   `www.xmind.works`). User quit Claude, ran the fix script, resumed,
   and connectivity was restored — verified by reaching the XMind
   "Recents" view and opening the `[draft] rosetta-stone-AI` map.

2. Learned the XMind editing surface. Switched from the default
   Mindmap (radial) view to the Outliner view (bottom-right toggle),
   which the user had already found easier for text editing than
   manipulating nodes directly on the radial canvas.

3. Diagnosed and fixed a formatting defect: nearly every topic title in
   the map carried a stray leading bullet-dot prefix (`• ` or `◦ `)
   baked directly into the title text (not a list-style artifact).
   Confirmed by directly editing one node ("Model") via
   click+Home+Shift+Right+Delete in the Outliner text editor — this
   manual method is 100% reliable but node-by-node, which the user
   flagged as "painfully long" at map scale (~122 affected titles).

4. Worked out a reliable method for using XMind's built-in Copilot
   (via its in-app chat panel, driven through Claude-in-Chrome) to do
   this at scale instead of by hand. Full detail and the concrete
   prompt pattern are in `Topics/xmind-copilot-collaboration.md`; in
   short:
   - A single node passed to Copilot worked correctly and cheaply.
   - Asking Copilot to fix the *entire map* in one ~104-operation batch
     failed: it confused the two different bullet glyphs in use,
     mismatched titles against hidden leading-space noise in its own
     tool output, and ultimately made almost no real edits despite
     reporting "Done." twice — a real trap, since the chat transcript
     alone looked like success.
   - The reliable pattern: scope each Copilot request to one
     branch/parent at a time (~7-15 topics), and hand it the *exact
     literal old-text titles* to match rather than asking it to
     self-identify what needs fixing. Every batch scoped this way
     succeeded in a single clean pass, at a fraction of the credit cost
     of the failed whole-map attempt.
   - Also discovered a UI quirk worth remembering: the Copilot chat
     input submits on a bare Enter (not Shift+Enter-for-newline), so a
     prompt typed with real embedded newlines gets silently truncated
     mid-message. Multi-item batch prompts need to be composed as a
     single line, not a multi-line numbered list.

5. Applied that method across the whole map in 5 scoped batches
   (Model's 9 direct children; Application's 9 children; Model
   (API-service)/(Edge)/(RAG) plus Model (RAG)'s 9 children; and the 7
   remaining top-level branches — Agent, Tool, Retrieval, Memory,
   Security, Deployment Context Notes, Cross-cutting fields — plus
   their own children, 63 topics in that last batch alone). Verified
   the full result after a page reload by pulling the live outline text
   and confirming zero remaining bullet-glyph prefixes anywhere in the
   map, cross-checked against Copilot's own per-batch diff logs. No
   other content (colors, tree structure, body/note text) was altered
   — confirmed line-by-line against the diffs.

METHOD DISCOVERIES
--------------------
- Claude-in-Chrome's domain lockout is per-session and recurs every
  session until fixed upstream; the mitigation is procedural (always
  navigate straight to the confirmed-working target domain first), not
  a one-time fix. See Topics/claude-in-chrome-connectivity.md.
- For AI-copilot-driven bulk edits generally (not just XMind): don't
  hand a copilot an open-ended "fix everything" instruction at scale
  and trust its own "Done." — scope requests narrowly, supply exact
  literal targets rather than asking it to identify them, and verify
  against a structured diff/log rather than prose output. This
  generalizes beyond XMind to any similar copilot-in-a-tool workflow
  this project might use later.
