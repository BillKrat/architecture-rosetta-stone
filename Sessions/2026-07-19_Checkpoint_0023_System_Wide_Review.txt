OBJECTIVES COMPLETED
---------------------

1. **Found and fixed a real disaster-recovery bug, not just a
   documentation gap.** `sync-skills` derives its backup namespace from
   `basename($REPO_ROOT)` at run time — it is not one shared bucket.
   Before this pass, only an `AI/` namespace existed in `ai-skills`
   (matching the workspace root's own basename). A restore run from
   inside `architecture-rosetta-stone`, `vs-mcp-bridge`, or `BlogAI`
   would have looked for a namespace matching that repo's name, found
   nothing, and silently restored zero skills — the exact kind of
   failure a disaster-recovery system exists to prevent, and one that
   would only surface at the worst possible time (an actual fresh-machine
   restore). Fixed by populating `architecture-rosetta-stone/`,
   `vs-mcp-bridge/`, and `BlogAI/` as their own namespaces in
   `ai-skills`, matching what the skill's own documented algorithm
   actually expects.

2. Fixed a real, already-spreading typo: `AGENTS.md`'s "Reuse Standard"
   listed `architecture-resetta-stone` and `vs-ms-bridge` — wrong in
   four of five copies (only the just-edited `ai-skills` copy had been
   corrected). Propagated the fix to `architecture-rosetta-stone`,
   `vs-mcp-bridge`, and `BlogAI` directly (each its own commit, each
   pushed).

3. Found `BlogAI` was missing entirely from the "Known Local
   Installations" list added in Checkpoint 0022 — it has the identical
   5-skill setup and its own `AGENTS.md` matching the others, but had
   never been checked. Added it everywhere the other three repos are
   already named.

4. Added a workspace-root `AGENTS.md` — none existed, despite this
   session routinely operating with the workspace root itself as the
   working directory (not inside any single repo), meaning routing
   guidance wasn't being loaded there at all.

5. Found and documented a real structural mismatch: `vs-mcp-bridge` has
   neither `.project-context/` nor the legacy `INDEX.md`+`Sessions/`+
   `Topics/` layout `lets-code`/`update-context` expect — it has its
   own, already-working, more mature convention (`AI_START.md` +
   `SolutionFolder/docs/session-handoffs/`). Without a guard, a future
   session could get steered toward `initialize-context` and create a
   redundant, conflicting structure. Added an explicit
   `vs-mcp-bridge-Specific Note` section to that repo's own `AGENTS.md`
   rather than changing the shared skill logic (lower risk — doesn't
   touch what already works correctly in the other repos).

6. Checked `BlogAI` for the same risk — same missing layout, but no
   clearly established alternative (`docs/session-handoffs/` has only
   one entry, from 2026-04-19, and the repo's own `README.md` says the
   project is "on hold"). Added an appropriately-hedged note there
   instead of asserting a convention that isn't actually established.

7. Corrected my own just-written claim that `AGENTS.md` should be
   byte-identical across all five locations — it no longer is, by
   design, now that `vs-mcp-bridge` and `BlogAI` carry their own
   append-only sections. Narrowed the "keep in sync" instruction to the
   three shared sections specifically (Skill Routing, Shared Skill
   Behavior, Reuse Standard).

METHOD DISCOVERIES
--------------------
- A backup system can look completely fine from the outside (files
  exist, README reads sensibly, skills are present locally) while
  containing a restore-time failure that only a genuine trace-through of
  the actual algorithm — not just reading its description — would catch.
  Reading `sync-skills`' own `SKILL.md` line by line, specifically the
  `PROJECT_NAMESPACE="$(basename "$REPO_ROOT")"` line, is what surfaced
  the real bug; the higher-level README and AGENTS.md descriptions had
  both been consistent with a wrong mental model of how it worked.
- Fixing a shared file's content is not the same as fixing its
  consistency — propagating the `AGENTS.md` typo fix required touching
  four separate repos individually, since nothing currently automates
  keeping `AGENTS.md` in sync the way `sync-skills` automates
  `.claude/skills/` itself. Worth remembering as a standing manual
  discipline, not a one-time fix.
