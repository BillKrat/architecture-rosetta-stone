OBJECTIVES COMPLETED
---------------------
1. Diagnosed the actual gap: `lets-code` never had a formally specified
   repo-picker step at all. Multi-repo selection had been ad hoc,
   improvised per-session by whichever assistant turn noticed the
   workspace root wasn't a git repo — which is why it was fragile
   enough to regress to plain prose without anyone deciding to remove
   it.

2. Added a durable Step 0 branch to `lets-code/SKILL.md`: when
   `git rev-parse --show-toplevel` fails at the current directory
   (multi-repo workspace root case), scan `repos/` (or direct
   subdirectories if no `repos/` folder) for `.git` entries. Zero
   candidates -> say so and stop. Exactly one -> use it directly, no
   prompt. Multiple -> call the `AskUserQuestion` tool with one
   radio-button option per repo, explicitly forbidding a plain-text
   substitute.

3. Included an inline, self-referential warning in the skill text
   itself: if a future edit is tempted to fold the `AskUserQuestion`
   call back into prose "for simplicity," that is the regression to
   avoid, not an acceptable trim. This is the same durability lesson as
   Checkpoint 0025's Active Thread field — a behavior that only lives
   in an assistant's improvised judgment doesn't survive revision;
   promoting it to an explicit, named skill instruction does.

4. Replaced the old `REPO_ROOT="$(git rev-parse --show-toplevel
   2>/dev/null || pwd)"` fallback, which silently used the non-repo
   workspace root itself as `REPO_ROOT` — the actual mechanism that let
   the picker regress invisibly, since falling back to `pwd` never
   surfaced a decision point at all.

5. Propagated the edited `lets-code/SKILL.md` to all 9 physical
   locations (workspace root, `architecture-rosetta-stone`, `BlogAI`,
   `vs-mcp-bridge`, and all 4 `ai-skills` backup namespaces plus its own
   `.claude/skills` copy), verified byte-identical via `diff`.

6. As a side effect, this also caught up `ai-skills/.claude/skills/
   lets-code/SKILL.md`, which had silently fallen behind on Checkpoint
   0025's Active Thread change — found by the same byte-identical check
   before making this session's edit.

METHOD DISCOVERIES
--------------------
- A UI behavior an assistant produces by its own in-the-moment judgment
  (deciding to call `AskUserQuestion` because multiple repos were
  found) is exactly as durable as that judgment being made fresh each
  session — it isn't backed by any instruction, so a revision pass can
  drop it without anyone noticing it was ever a deliberate behavior.
  Same lesson as Checkpoint 0025's Active Thread fix, applied to a
  different symptom: promote the behavior to a named, explicit skill
  step, and add a standing note warning against the specific
  simplification that caused the regression.
- Bill's framing this session: iterating through "deleting session and
  starting fresh one to work out the kinks" from his own earlier
  cleanup request — consistent with the cross-repo infra churn visible
  across Checkpoints 0021-0026 all landing on 2026-07-19.
